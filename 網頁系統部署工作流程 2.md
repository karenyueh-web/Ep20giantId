## 📨 資源申請與變數配置

### 步驟 1 — 資源申請（人工作業）
前置條件：已確認專案名稱與目標環境。
*💡 AI 助手可協助維運人員依據以下規格，自動撰寫正式的資源申請信件內容。*

* **a. Domain**: `${DOMAIN}`
  * **用途**：ALB HTTPS 接聽規則的 Host-based routing 條件。
  * **確認事項**：DNS 是否由 Infra Team 統一於內部 DNS 建立 CNAME → ALB DNS Name。
* **b. AWS Console 存取權限**:
  * **帳號類型**：個人 IAM 帳號（非 root）。
  * **所需權限範圍**：IT-Project Dev（ECS、ECR、CloudWatch Logs、ELBv2、SSM）。
  * **確認事項**：是否需要啟用 MFA。
* **c. ECR 儲存庫**: `${ECR_REPO}`
  * **確認事項**：
    * Image tag mutability 建議設為 `IMMUTABLE`（正式環境）或 `MUTABLE`（開發環境）。
    * Lifecycle policy 是否需要設定（例如保留最近 10 個 image）。
* **d. ECS 角色**: `${ECS_ROLE}`
  * **用途**：授予容器內部應用程式（程式碼）存取其他 AWS 服務的權限，以及 ECS 容器代理程式（ECS Agent）執行 AWS API 調用的權限。
  * **確認事項**：測試該角色是否有足夠權限執行系統需求。
* **e. OIDC 角色**: `${OIDC_ROLE}`
  * **用途**：允許此 GitLab repository 的指定 branch 透過 OIDC 取得 AWS 臨時憑證。
  * **設定要求**：需在 IAM Role `gitlab-ci-role` 的 Trust Policy 中新增 Condition：
    ```json
    "StringLike": {
      "gitlab.com:sub": "project_path:<GitLab-Group>/\${PROJECT_NAME}:ref_type:branch:ref:<branch-name>"
    }
    ```
    *範例：* `"gitlab.com:sub": "project_path:giant-group/giant-audit:ref_type:branch:ref:main"`
  * **確認事項**：
    * 確認 OIDC Provider (`gitlab.com`) 已在 AWS IAM Identity Providers 中註冊。
    * 確認 Role 的 Permission Policy 至少包含：`ecr:*`, `ecs:*`, `logs:*`, `ssm:GetParameters`, `elasticloadbalancing:*`（依最小權限原則調整）。
* **f. RDS 資料庫**:
  * **確認事項**：為 AWS RDS 服務，需取得明確的 hostname、port、user、password 的資訊。
* **g. 網路與架構環境資訊確認**:
  * **用途**：配置 ECS 服務網路連線與安全隔離。
  * **確認事項**：請 Infra 團隊提供專案部署目標環境的 **VPC_ID**、**SUBNET_1**、**SUBNET_2**（需為具備 NAT Gateway 的 Private Subnet）以及 **SG_ID**（ECS 專用安全組 ID）。

**📦 產出物**：取得 Domain、IAM 帳號、ECR repo、ECS 角色、OIDC 角色、資料庫已建立的確認，以及專案環境的 **VPC ID**、**Private Subnet IDs (至少兩組)**、**ECS Security Group ID**。

## 🛠️ AWS 資源建立（AI 可協助執行）

前置條件：
- 已取得步驟 1 的 IAM 帳號存取金鑰（Access Key + Secret Key）。
- 已執行 `aws configure` 設定 credentials 與 region。
- 已確認帳號擁有所需權限。

### 3a. 建立 SSM Parameter Store 環境變數
**用途**：讓 ECS Task 在啟動時透過 `valueFrom` 從 SSM 安全取得設定值。
**命名格式**：`/${ENV}/${PROJECT_NAME}/{參數名稱}`
**範例**：
- `/${ENV}/${PROJECT_NAME}/DATABASE_URL`
- `/${ENV}/${PROJECT_NAME}/API_KEY`

```bash
# AWS CLI 範例
aws ssm put-parameter \
  --name "/\${ENV}/\${PROJECT_NAME}/DATABASE_URL" \
  --value "postgres://user:pass@host:5432/db" \
  --type "SecureString" \
  --overwrite
```
* **注意事項**：
  * 機敏資料使用 `SecureString`（KMS 加密），非機敏資料可用 `String`。
  * Task Execution Role 須有 `ssm:GetParameters` 權限及對應 KMS decrypt 權限。

---

### 3b. 建立 CloudWatch Log Group
```bash
aws logs create-log-group --log-group-name "\${LOG_GROUP}"
aws logs put-retention-policy --log-group-name "\${LOG_GROUP}" --retention-in-days 30
```
* **注意事項**：
  * retention 建議依環境設定：dev = 7天, uat = 30天, prod = 90天。
  * 確認 Task Execution Role 有 `logs:CreateLogStream`, `logs:PutLogEvents` 權限。

---

### 3c. 建立 Target Group
```bash
aws elbv2 create-target-group \
  --name "\${TARGET_GROUP}" \
  --protocol HTTP \
  --port \${CONTAINER_PORT} \
  --vpc-id \${VPC_ID} \
  --target-type ip \
  --health-check-path "/health" \
  --health-check-interval-seconds 30 \
  --healthy-threshold-count 2 \
  --unhealthy-threshold-count 3
```
* **注意事項**：
  * health-check-path 請確認應用程式有實作健康檢查端點（例如 `/health` 回傳 200）。
  * 若無健康檢查端點，可暫用 `"/"` 但建議盡早補上。
  * target-type 必須為 `ip`（Fargate 需求）。

---

### 3d. 設定 ALB Listener Rule（HTTPS:443）
前置條件：確認 ALB「`${ALB_NAME}`」已存在且有 HTTPS:443 Listener。

```bash
# 1. 透過 ALB_NAME 先行查詢取得 ALB_ARN
ALB_ARN=\$(aws elbv2 describe-load-balancers \
  --names "\${ALB_NAME}" \
  --query "LoadBalancers.LoadBalancerArn" --output text)

# 2. 透過動態取得的 ALB_ARN 查詢 Listener ARN
LISTENER_ARN=\$(aws elbv2 describe-listeners \
  --load-balancer-arn "\$ALB_ARN" \
  --query "Listeners[?Port==\`\${LISTENER_PORT}\`].ListenerArn" --output text)

# 3. 動態查詢目前該 Listener 底下規則的最大 Priority 數值，並自動增加 10
# (若查無現有數字規則，則預設從 10 開始)
MAX_PRIORITY=\$(aws elbv2 describe-rules \
  --listener-arn "\$LISTENER_ARN" \
  --query "Rules[?Priority!=\`default\`].Priority | max(@)" --output text)

if [ "\$MAX_PRIORITY" = "None" ] || [ -z "\$MAX_PRIORITY" ]; then
  NEW_PRIORITY=10
else
  NEW_PRIORITY=\$((MAX_PRIORITY + 10))
fi

# 4. 新增規則 (帶入自動算出的 NEW_PRIORITY)
aws elbv2 create-rule \
  --listener-arn "\$LISTENER_ARN" \
  --priority \$NEW_PRIORITY \
  --conditions '[{"Field":"host-header","Values":["\${DOMAIN}"]}]' \
  --actions '[{"Type":"forward","TargetGroupArn":"<TARGET_GROUP_ARN>"}]'
```
* **注意事項**：
  * 腳本已自動處理 Priority 遞增邏輯（取現有最大值 + 10），避免人工比對與衝突。
  * 確認 ALB 的 Security Group 允許 inbound 443。
  * 確認 ACM 憑證涵蓋 `${DOMAIN}`（萬用字元 `*.${ENV}.giantgroup.local` 或個別申請）。

---

### 3e. 建立 ECS Task Definition
建立 `task-def.json` 檔案：
```json
{
  "family": "\${TASK_DEF}",
  "networkMode": "awsvpc",
  "requiresCompatibilities": ["FARGATE"],
  "cpu": "\${CPU}",
  "memory": "\${MEMORY}",
  "executionRoleArn": "\${EXECUTION-ROLE-ARN}",
  "taskRoleArn": "\${TASK-ROLE-ARN}",
  "containerDefinitions": [
    {
      "name": "\${PROJECT_NAME}",
      "image": "\${DOCKER_IMAGE}",
      "portMappings": [
        {
          "containerPort": \${CONTAINER_PORT},
          "protocol": "tcp"
        }
      ],
      "logConfiguration": {
        "logDriver": "awslogs",
        "options": {
          "awslogs-group": "\${LOG_GROUP}",
          "awslogs-region": "\${AWS_REGION}",
          "awslogs-stream-prefix": "ecs"
        }
      ],
      "secrets": [
        {
          "name": "DATABASE_URL",
          "valueFrom": "arn:aws:ssm:\text{\$}AWS_REGION:\text{\$}AWS_ACCOUNT_ID:parameter/\text{\$}ENV/\text{\$}PROJECT_NAME/DATABASE_URL"
        }
      ],
      "essential": true
    }
  ]
}
```

```bash
# 註冊指令
aws ecs register-task-definition --cli-input-json file://task-def.json
```
* **注意事項**：
  * `executionRoleArn`：用於拉取 image、讀取 SSM、寫入 Log（系統層）。
  * `taskRoleArn`：用於容器內應用程式存取其他 AWS 服務（應用層）。
  * CPU/Memory 有效組合參考：256/512, 256/1024, 512/1024, 1024/2048 等。

---

### 3f. 建立 ECS Cluster 與 Service
```bash
# 建立 Cluster
aws ecs create-cluster --cluster-name "\${CLUSTER}"

# 建立 Service
aws ecs create-service \
  --cluster "\${CLUSTER}" \
  --service-name "\${SERVICE}" \
  --task-definition "\${TASK_DEF}" \
  --desired-count 1 \
  --launch-type FARGATE \
  --network-configuration "awsvpcConfiguration={subnets=[\${SUBNET_1},\text{\$}SUBNET_2],securityGroups=[\text{\$}\)SG_ID}],assignPublicIp=DISABLED}" \
  --load-balancers "targetGroupArn=<TARGET_GROUP_ARN>,containerName=\${PROJECT_NAME},containerPort=\${CONTAINER_PORT}"
```
* **注意事項**：
  * Subnets 選擇 Private Subnet（透過 NAT Gateway 出網路拉取 ECR image）。
  * Security Group 需允許：
    * Inbound：來自 ALB SG 的 `${CONTAINER_PORT}`。
    * Outbound：443（ECR、SSM、CloudWatch）。
  * `assignPublicIp` 設為 `DISABLED`（走 private subnet + NAT）。
  * `desired-count` 初始可設 1，穩定後再考慮 auto-scaling。

## 🔍 4. 部署驗證與排查

**驗證網址**：`https://${DOMAIN}`

### ✅ 檢查清單 (Checklist)
- [ ] DNS 解析正確（`nslookup ${DOMAIN}` → ALB IP）
- [ ] HTTPS 憑證有效（瀏覽器無警告）
- [ ] 頁面正常回應（HTTP 200）
- [ ] Health Check 通過（Target Group 顯示 healthy）
- [ ] CloudWatch Logs 有容器啟動日誌
- [ ] 應用程式功能正常

### 🚨 常見問題排查

| 現象 | 可能原因 | 建議排查方向 |
| :--- | :--- | :--- |
| **502 Bad Gateway** | Container 尚未啟動完成 / Port 對應錯誤 | 檢查應用程式監聽 Port 是否與 `${CONTAINER_PORT}` 一致。 |
| **503 Service Unavailable** | Target Group 無 healthy target | 檢查應用程式是否成功對外提供服務且健康檢查路由正確。 |
| **Connection Timeout** | Security Group 未放行 / Subnet 路由問題 | 檢查 ECS 與 ALB 之間的安全組 Inbound/Outbound 規則。 |
| **Task 持續重啟** | 環境變數缺失 / 應用程式 crash / OOM | 透過 CloudWatch 檢查容器崩潰前的錯誤日誌。 |
| **Image pull 失敗** | ECR 權限不足 / NAT Gateway 問題 | 檢查 Task Execution Role 權限，以及 Subnet 能否正常出網。 |

#### 💻 排查指令庫
```bash
# 查看服務狀態與事件歷史
